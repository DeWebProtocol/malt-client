# MALT v0.0.5 Application Migration

MALT v0.0.6 made `DeWebProtocol/malt` an SDK-only core. This matrix records the
disposition of application and tool code that previously lived in the core
repository at v0.0.5.

| Former capability | Current owner | Status |
| --- | --- | --- |
| `malt add` with staged hybrid MALT UnixFS | malt-client | migrated to generic gateway mutation/CAS APIs; former v0.0.5 flat/hierarchical aliases are no longer exposed |
| `malt add --target merkle-dag` | malt-client | migrated with balanced/trickle files, basic/HAMT/adaptive directories, and UnixFS symlinks that are encoded rather than followed |
| `.gitignore`, `.maltignore`, explicit ignore files | malt-client | migrated and tested for both targets |
| resolve and ProofList verification | malt-client | migrated to v0.0.6 resolve contract and local verifier |
| native UnixFS `Stat`, file/range reads, and list payload reads | malt-client | migrated as a transport-neutral verified facade over generic resolve/read/CAS; raw, manifest, and range bytes are CID-bound locally |
| UnixFS `RemovePath` | malt-client | migrated as immutable candidate-root materialization; never auto-accepted |
| public Go gateway transport | malt-client | migrated to `github.com/dewebprotocol/malt-client/transport`; narrow native/CAS/diagnostic ports expose no internal DTOs |
| accepted/candidate root policy | malt-client | added; gateway results never auto-promote |
| reusable client application orchestration | malt-client | shared `application` use cases now compose accepted-root selection, UnixFS add/read/remove, candidate record/accept, and Merkle DAG import/read for CLI and daemon adapters |
| local daemon | malt-client | redefined as trusted-root control plane, with private Unix socket on Unix and access-controlled named pipe on Windows |
| reference executor and HTTP routes | gateway | replaced by embedded generic resolve/read/apply/CAS service |
| ArcTable, KV, and CAS persistence | gateway | migrated; not imported by client or core |
| standalone CAS process | none | deliberately removed; gateway owns product CAS |
| evaluator and benchmark suites | [malt-evaluation](https://github.com/DeWebProtocol/malt-evaluation) | former v0.0.5 evaluator retired from the active tree and recoverable from the [immutable standalone import commit](https://github.com/DeWebProtocol/malt-evaluation/commit/774854b5d90c1ba5d57c3d74619dd97a78fd9dcf) and Git history; active workloads live at the repository root |
| Merkle DAG gateway compatibility reads | gateway + malt-client | gateway traverses UnixFS and returns CID-bound blocks; client independently hashes and replays them under a distinct non-MALT profile |
| profile-specific transport escape hatch | none | arbitrary route/body transport was removed; Merkle DAG compatibility uses fixed resolve/read capabilities |
| generic IPLD `ParseBlock`/link helpers | malt-client | restored as `merkledag/ipld` with raw, DAG-PB, DAG-CBOR, DAG-JSON, and legacy JSON codecs |
| large native UnixFS writer | malt-client | public writer supports sized streaming without buffering the whole file; unknown-size streams spool to a temporary file before immutable materialization |
| CAS batch transport and diagnostic metrics | malt-client | public bounded batch methods and typed monotonic Gateway metrics snapshot |

## Compatibility target semantics

`malt add` supports two explicit targets:

- `--target malt`: constructs MALT-authenticated application objects, returns
  a candidate MALT root, and can participate in accepted-root policy.
- `--target merkle-dag`: constructs IPFS-compatible UnixFS blocks, stores them
  in CAS, and returns the Merkle DAG root CID.

The second mode is interoperability support. It does not wrap a Merkle DAG root
in MALT evidence and does not make it eligible for `malt root accept`.

## Remaining pre-release work

- keep the gateway-owned cross-repository product E2E in CI;
- publish fresh measurements from the root evaluator's current-product
  deterministic read matrix and logical-storage accounting;
- define the first malt-client release and compatibility policy only after
  these checks remain stable.

The standalone v0.0.5 CAS process was not copied because its valid storage
behavior is now provided by Gateway CAS, including bounded batch operations.
Likewise, the old graph manager maps to Gateway execution scopes. These are
behavior migrations, not missing source directories.
